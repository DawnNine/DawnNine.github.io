Java反序列化Commons-Collection篇CC3链

CC3的入口为TemplatesImpl.defineClass()，环境采用CC1的环境版本都是commons-collections3.2.1，我的版本可能有点低1.8u64，跟网上的师傅源码中存在出入正常现象。如果不会CC1推荐师傅先看CC1，CC3是基于CC1的绕过和扩展。

对比CC1和CC2，CC3的利用方式是代码执行，CC1和CC2是通过反射的方式来调用Runtime.exec()，通过Runtime.exec()来执行命令，CC3利用的是TemplatesImpl.defineClass()，进行远程加载一个类，这个有点类似于JND注入。都是加载一个远端的类，进行反序列化执行恶意代码。

CC3链分析

入口点分析

TransletClassLoader.defineClass()方法是将一段符合 JVM 规范的字节码（byte[]）转换成一个可用的 java.lang.Class 对象，并注册到 JVM 中。

TransletClassLoader.defineTransletClasses()方法调用了defineClass方法，要求_bytecodes不能为空，方法属性是private方法接着往下找

_class[i] = loader.defineClass(_bytecodes[i]);

TransletClassLoader.getTransletInstance()方法调用了defineTransletClasses方法，两个if判断_name和_class参数不能为空，我们希望它调用defineTransletClasses，所以_class不能赋值。方法属性是private方法接着往下找。

if (_class == null) defineTransletClasses();

TransletClassLoader.newTransformer方法调用了getTransletInstance()并且这个是public方法。

transformer = new TransformerImpl(getTransletInstance(), _outputProperties, _indentNumber, _tfactory);

开始构建参数

在getTransletInstance方法中，需要满足_name不为空，_class它默认为空。而走到defineTransletClasses方法中，_bytecodes不能为空(我的版本比较低1.8u64不存下面这一块)。

return new TransletClassLoader(ObjectFactory.findClassLoader(),_tfactory.getExternalExtensionsMap());

高版本需要从_tfactory中调用getExternalExtensionsMap方法，在readObject方法中对这个属性赋值了。所以在序列化操作中不用管这个属性了，而我们测试的时候需要给它赋值。

总结

CC1 和 CC3 前半段（触发点）相同，都是利用 AnnotationInvocationHandler.readObject() → LazyMap.get() 触发 Transformer.transform()； 后半段不同：CC1 使用 InvokerTransformer 反射调用方法执行命令，CC3 使用 TemplatesImpl + 字节码动态加载执行命令，从而绕过对 InvokerTransformer`的检测。
