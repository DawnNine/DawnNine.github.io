Java Web程序运行流程

​ 一个基本的java web程序是由前端、后端、中间件（既应用服务器）、数据库。用一个最简单的流程来说，用户在前端发送请求（如通过浏览器），请求首先到达 Web 服务器 / 应用服务器（例如内嵌的 Tomcat），该服务器接收到 HTTP 请求后，根据 URL 路由将其分发给对应的 后端业务代码进行处理。后端代码根据业务逻辑进行计算，并通过数据访问层对数据库进行增删改查操作。操作完成后，后端将结果封装成响应（如 JSON 或 HTML），由应用服务器通过 HTTP 协议返回给前端。前端接收到响应后，更新页面或展示数据。

从请求入口到业务逻辑：审计的关键路径

​ 从上面流程可以进行思考，java后端代码是如何处理前端传入的查询指令，举一个简单的例子，我要根据用户ID来查对应的用户名。http://localhost:8080/user?id=1 ，看返回结果先思考为什么应用服务器能够把指令准确的传递给java后端，java后端代码为什么能够准确的对数据库查询，并且页面返回结果。

​ 由 Tomcat 应用服务器的 Connector 接收请求，获取请求路径。Mapper 组件根据该路径进行匹配，查找对应的目标 Servlet。由于 Mapper 的路径解析较为宽松，例如 /;/user 也可能被匹配到 /user 对应的 Servlet。匹配成功后，Tomcat 会根据该 Servlet 的映射路径，在已注册的 Filter 列表中查找所有匹配的过滤器。如果存在匹配的 Filter，则先执行过滤器逻辑。若过滤器中判断条件成立（如路径为 /user），则返回 403 拦截请求。若未被拦截，请求将继续传递，最终到达 Servlet 或 Controller 执行具体业务。

安全实践与防护思路

​ 在全面了解了 Java Web 请求的完整流程后，我们发现漏洞与功能点密切相关。很多漏洞的产生，往往源于功能点开发不够严谨或安全意识不足。深入理解每个功能点的实现逻辑，才能准确发现潜在的风险并进行有效的审计和防护。
